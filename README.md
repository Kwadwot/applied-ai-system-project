# 🎮 Number Guessing Game with RAG Game Coach

## Original Project

**Project name:** Game Glitch Investigator (Modules 1–3)

The original project was a deliberately broken Streamlit number guessing game where the goal was to find and fix multiple AI-generated bugs — including a resetting secret number, inverted higher/lower hints, incorrect difficulty ranges, and a broken new-game reset. After debugging and refactoring, the game became a fully functional number guessing app with difficulty levels, attempt limits, a scoring system, and guess history tracked via Streamlit session state.

---

## Title and Summary

**Number Guessing Game with RAG Game Coach** extends the debugged game from Modules 1–3 by adding an AI-powered coaching feature. When a player is stuck, they can click "Ask the Coach" to receive a personalized strategy hint generated by Gemini 2.0 Flash. The hint is grounded in a curated knowledge base of number-guessing strategies retrieved at query time using ChromaDB — making this a practical example of Retrieval-Augmented Generation (RAG). The coaching system matters because it demonstrates how to connect a vector store, a local embedding model, and an LLM into a working AI pipeline layered on top of an existing application.

---

## Architecture Overview

The system has two parallel flows that share the same Streamlit UI:

**Game flow:** The player types a guess, which is validated by `parse_guess()`, evaluated by `check_guess()`, and scored by `update_score()` — all in `logic_utils.py`. Results are written to Streamlit session state and displayed on screen.

**RAG coach flow:** When the player clicks "Ask the Coach," the current game state (difficulty, range, guess history, last feedback) is used to build a retrieval query. ChromaDB searches an in-memory collection of 10 strategy documents — embedded once at startup using sentence-transformers — and returns the 3 most relevant. Those documents plus the game context are sent to Gemini 2.0 Flash, which generates a short, personalised strategy hint displayed in the sidebar.

Human evaluation happens at two points: the player judges whether the coaching hints are useful, and a developer can inspect the raw game state via the "Developer Debug Info" expander. Automated evaluation of the core game logic is handled by `pytest` in `tests/test_game_logic.py`.

## 🛠️ Setup Instructions

1. **Clone the repository and enter the project directory.**

2. **Add your Gemini API key.** Open the `.env` file and replace the placeholder:
   ```
   GEMINI_API_KEY=your_key_here
   ```

3. **Install dependencies.**
   ```bash
   pip install -r requirements.txt
   ```
   > Note: `sentence-transformers` will download the `all-MiniLM-L6-v2` model (~90 MB) on first run. This is a one-time download.

4. **Run the app.**
   ```bash
   streamlit run app.py
   ```

5. **Run the tests** (optional).
   ```bash
   pytest
   ```

## Design Decisions

**ChromaDB EphemeralClient (in-memory, no server)**
ChromaDB can run as a persistent on-disk store or as a separate server, but both add setup burden for anyone cloning the project. EphemeralClient runs entirely inside the Python process — no server to start, no folder to manage. The knowledge base is small (10 documents), so re-embedding on each startup takes only a few seconds.

**sentence-transformers for embeddings (local, no API cost)**
Generating embeddings locally with `all-MiniLM-L6-v2` means retrieval has no API cost and no network dependency. The trade-off is a one-time ~90 MB model download and a slightly slower cold start compared to using a hosted embedding API.

**Gemini 2.0 Flash for generation**
Gemini 2.0 Flash is a fast, low-cost model well-suited for short hint generation. The trade-off versus a larger model is that hints may be less nuanced, but for 2–3 sentence coaching responses the quality is sufficient and latency stays low.

**`st.cache_resource` for the ChromaDB collection**
The collection is embedded once and cached for the lifetime of the Streamlit process. This means the embedding cost is paid only on first use, and every subsequent "Ask the Coach" click hits an already-loaded in-memory store.

**Integer-only history passed to the LLM**
Invalid guesses (non-numeric strings) are stored in session state for display purposes but are filtered out before being included in the coach prompt. This prevents a user from injecting arbitrary text into the LLM prompt by typing something other than a number.

**python-dotenv for API key management**
The `.env` file is already covered by `.gitignore`, so the key is never committed. `load_dotenv()` in `rag_coach.py` loads it automatically at startup, so the user does not need to set an environment variable in the terminal each session.
